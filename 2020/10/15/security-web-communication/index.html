WEB安全通信

Public and Private KEY Cryptography

方案一：私钥加密（对称加密算法）：发送者和接收者之间共享私钥

如果你想要和某个人进行加密通信，那么我们就可以将私钥给他们。但是这里会有问题：私钥丢失、私钥被窃取都会导致严重问题。

方案二：私钥公钥一起使用（非对称加密算法）

RAS是第一个实际使用的公钥/私钥加密算法，被广泛数据通讯上。RAS安全性极高，黑客们很难从公钥上计算得到私钥，现在能破解到达仅为769位数字。而现在大部分使用的是1024位数字，甚至2048。

Cryptographic Hash Functions

  • 一个hash函数映射一组值到一定范围的数字上。

  • 给定一个数据值X（X可以是一个文字或者一个文件），H仅当能够将任一值X映射到唯一对应的值域，那么它才是一个cryptographic hash函数。H也被称为数据X的数字签名(message digest or digital signature)。

  • 两个著名的数字签名hash函数是MD5和SHA

real-time需求

公钥/私钥方法并不适合一般性的需求。

  • RSA算法只能分段加密，因为其只能加密比其密钥少11bytes的内容，而且RSA加密的计算都非常复杂，这样并不适合web高频率的通信。
  • 服务端私钥加密内容可以被所有拿了公钥的人读取，所以我们需要对称加密

web通信现在采用的是公钥/私钥+私钥两种方式结合进行通信。bulk cipher就是一种等同加密算法（即要用同一个密钥加密和解密），优点在于其速度非常快。

  • 常用的bulk cipher：RC2, RC4-40, RC4-56, DES40-CBC

数字签名

  • A message digest is the number produced by applying a cryptographic hash function to a message

    • The message digest is included along with the message; here are the steps that are followed:
      1. Sender produces a message digest using a known hashing algorithm
      2. Message digest is encrypted and sent with the message
      3. Receiver decrypts the digest and then computes the message digest from the actual message to make sure they are identical

  • For greater security, the message can also beencrypted

  • Systems combining public key cryptography and message digests are called digital signatures

证书认证

怎么从web交流的开始就可以确认这是一个正确的网站呢？发布出去的公钥很可能被恶意篡改，因为有的人直接造一个钓鱼网站，然后骗取用户输入重要信息。

CA(certificate authority) 第三方权威认证机

CA会去核实对应的clinet/server站点，核实成功后就会颁发数字证书。浏览器一般会预先安装一系列CA证书。 CA证书中包含的是：证书版本信息、证书序列号、颁发机构，被认证的站点，有效时间，CA认证机构对证书内容的签名、站点公钥。

SSL

SSL(Secure Sockets Layer)是一个建立加密连接的协议，这个协议规定了使用认证和加密的数据传输。SSL最初被称为TLS(Transport Layer Security)。SSL协议是在HTTP和TCP之间的，因此SSL也可以加密其他的应用层，比如：FTP、NNTP。

SSL(443)特点：

  • 认证：使用数字证书认证双方
  • 保密：使用加密算法进行加密
  • 信息完整性：使用数字签名来保证数据没有被篡改。

SSL工作流程

1. 客户端发送一个https请求，并要求server验证自己
2. 服务器给用户发送数字证书（包含了公钥）来验证自己省份
3. 客户检查受信任的CA机构的根证书，并且确定其未过期。如果验证成功那么客户端会生成一个session key，并使用公钥对其加密然后发送这个加密信息给服务器。（验证过程：根证书逐步往下查找得到证书中CA公钥，直到找到了对应证书的中包含的公钥，我们就可以利用这个CA公钥对服务器传来的公钥的签名进行签名对照验证有效性，并且验证URL一致性）。
4. 服务器使用自己的私钥解密的session key（这个其实就是一个对称加密算法的key），返回一个确认信息，此时 安全通道建立完成。
5. 除了对称加密之外，还会使用数字签名来保证数据完整性。
